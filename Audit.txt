🏛️ "النظام البيئي الخالد" – الأرشيتيكتر النهائي للأوديت والتحليلات والبث الحي (الإصدار 11.0 – Cloudflare Native Final)
الهوية: نظام متكامل لإدارة الأحداث، التحليلات الفورية، والبث الحي، يعمل بتكلفة تشغيلية صفرية، على بنية تحتية سحابية بالكامل (Cloudflare + Upstash)، مع عزل تام للبيانات، وسرعة استجابة خارقة، وقابلية للتوسع إلى ما لا نهاية.

📜 الديباجة: فلسفة "الجبل الجليدي" (The Iceberg Philosophy)
في هذا النظام، نطبق مبدأ فصل البيانات حسب درجة الحرارة (Data Temperature Separation):

الطبقة السطحية (Hot Layer): البيانات النشطة (آخر 24-48 ساعة) المخزنة في D1، وهي الطبقة الوحيدة القابلة للكتابة والتحديث.

الطبقة المتوسطة (Warm Layer): البيانات المؤرشفة مؤقتاً (الـ Raw JSON) في R2، في انتظار المعالجة.

الطبقة العميقة (Cold Layer): الأرشيف التاريخي غير المحدود في R2 بصيغة Parquet، وهو الطبقة الوحيدة القابلة للقراءة والتحليل.

الهدف: إبقاء الطبقة السطحية (D1) خفيفة وسريعة وقابلة للكتابة دائماً، مع نقل كل الثقل إلى R2 حيث التخزين غير المحدود والمجاني.

🧱 المكونات المعمارية الأساسية (The Core Components)
1. قاعدة البيانات النشطة (D1 – The Operational DB)
الطبيعة: قاعدة بيانات SQLite مُدارة موزعة عالمياً من Cloudflare.

الهيكلية: تعتمد على جداول شهرية (audit_logs_YYYY_MM) لتجنب التفتيت (Fragmentation).

الدور: مسؤولة فقط عن عمليات الكتابة (INSERT) للأحداث الجديدة، وعمليات القراءة (SELECT) للـ Polling الحي.

سياسة الكتابة: الجدول هو Write-Only طوال الشهر. لا تحديثات (UPDATE)، ولا حذف (DELETE) للبيانات النشطة.

سياسة الاحتفاظ: مع بداية كل شهر جديد، يتم إنشاء جدول جديد. الجدول القديم يُحذف بالكامل عبر DROP TABLE بعد التأكد من أرشفة كامل بياناته إلى R2.

2. ذاكرة التخزين المؤقت والعلامات (Upstash Redis – The Cache & Index)
الدور الأول (البث الحي): تخزين last_event_timestamp لكل متجر. يستخدم لتقنين طلبات Polling ومنع الوصول غير الضروري إلى D1.

الدور الثاني (التجميع): تخزين last_processed_timestamp لكل متجر. يستخدم لتحديد الأحداث الجديدة فقط منذ آخر جولة تجميع (Idempotency Key).

الدور الثالث (حماية التكرار): تخزين معرفات الأحداث التي تمت معالجتها مؤقتاً لمنع التكرار (Deduplication) في حالة فشل الـ Worker وإعادة المحاولة.

السبب: الـ Redis أسرع بـ 100 مرة من D1 في عمليات القراءة والكتابة (لأنه In-Memory)، ويحسب ضمن الخطة المجانية لـ Upstash (10,000 عمليات/يوم).

3. جدولة المهام (Upstash QStash – The Scheduler)
الدور: وكيل الجدولة المركزية. بديل عن Cron Triggers المدفوعة في Cloudflare.

آلية العمل: يتم إعداد مهام دورية (Cron Jobs) في QStash لتوجيه طلبات HTTP إلى نقاط نهاية محددة في Cloudflare Workers في أوقات محددة:

كل ساعة (Hourly): طلب تجميع الأحداث الخام.

كل يوم (Daily): طلب تحويل الـ JSON إلى Parquet ودمج اليوم.

كل أسبوع (Weekly): طلب دمج الأيام في ملف أسبوعي.

آخر يوم في الشهر (Monthly): طلب حذف الجدول الشهري المنتهي صلاحيته.

السبب: QStash يوفر 10,000 طلب مجاني شهرياً، وهو أكثر من كافٍ لهذه المهام الدورية، كما أنه يوفر آلية إعادة محاولة مدمجة (Retry) في حال فشل الـ Worker.

4. وحدة المعالجة الحافة (Cloudflare Workers – The Compute Engine)
الطبيعة: بيئة تشغيل JavaScript/WebAssembly عند الحافة (Edge).

الأدوار الأربعة للـ Workers:

Worker الالتقاط (Capture): يستقبل الأحداث HTTP من التطبيق، يكتبها في D1، ويُحدّث Redis.
Worker البث الحي (Live): يستقبل طلبات Polling من المتصفح، ويتحقق من Redis، ويقرأ من D1 فقط عند الضرورة.
Worker التجميع الخام (Raw Aggregator): يُشغل بواسطة QStash كل ساعة، يقرأ الأحداث الجديدة من D1، ويكتبها كـ JSON في R2.
Worker التحويل الثقيل (Heavy Transformer): يُشغل بواسطة QStash يومياً/أسبوعياً، يشغل DuckDB-WASM داخل الـ Worker نفسه لتحويل JSON إلى Parquet مباشرة بين R2 و R2 (Streaming)، دون تحميل الملفات في ذاكرة الـ Worker.
5. مخزن الكائنات البارد (Cloudflare R2 – The Immortal Archive)
الطبيعة: تخزين كائنات (S3-compatible) بدون رسوم خروج (Zero Egress).

الهيكل (Hive Partitioning):

text
/audit-logs
  /store_{id}
    /raw/               # المنطقة المؤقتة (تُحذف يومياً)
      /year=YYYY/month=MM/day=DD/chunk_{HH}.json
    /parquet/           # المنطقة الدائمة (للأبد)
      /year=YYYY/month=MM/day=DD/chunk.parquet
      /year=YYYY/month=MM/day_summary.parquet
      /year=YYYY/month=MM/week_summary.parquet
      /year=YYYY/monthly_summary.parquet
سياسة الاحتفاظ: غير محدود. لا يتم حذف أي ملف Parquet أبداً.

🔄 تدفق البيانات التفصيلي (مرحلة بمرحلة)
المرحلة الأولى: الالتقاط (Capture Phase) – الزمن: < 10ms
يحدث حدث في المنصة (شراء، تحديث منتج، إضافة للسلة).

يقوم التطبيق (Server Action/API) بإرسال طلب POST إلى Worker الالتقاط.

يقوم Worker بإنشاء كائن الحدث وإضافة طابع زمني (Timestamp) ومعرف فريد (UUID).

الكتابة في D1: ينفذ INSERT INTO audit_logs_YYYY_MM (الجدول النشط) مع فهرسة على store_id و created_at.

تحديث Redis: ينفذ SET store:{id}:last_ts = {timestamp} في Upstash Redis (مدة صلاحية ساعة واحدة).

يعيد Worker استجابة 200 OK للتطبيق، وتنتهي العملية.

المرحلة الثانية: البث الحي (Live Feed Phase) – الزمن: < 5ms (مع Cache Hit)
يفتح التاجر لوحة التحكم (Dashboard).

يقوم المتصفح بفتح اتصال Polling وإرسال طلب GET إلى Worker البث الحي مع If-None-Match يحتوي على آخر timestamp معروف.

يقوم Worker بالاستعلام عن Upstash Redis: GET store:{id}:last_ts.

سيناريو (Cache Hit): إذا كانت القيمة في Redis مساوية للقيمة المرسلة، يرد Worker فوراً بـ 304 Not Modified (بدون أي استعلام على D1).

سيناريو (Cache Miss): إذا كانت القيمة مختلفة أو منتهية الصلاحية، يقوم Worker بالاستعلام عن D1: SELECT * FROM audit_logs_YYYY_MM WHERE store_id = X AND created_at > Y ORDER BY created_at DESC LIMIT 50. يعيد النتائج مع 200 OK ويُحدّث Redis.

توفير الموارد: في سيناريو الخمول (لا توجد أحداث جديدة)، لا يتم لمس D1 على الإطلاق، مما يحافظ على حصة الـ Reads المجانية.

المرحلة الثالثة: التجميع الخام (Raw Aggregation Phase) – تُشغّل كل ساعة
يُطلق QStash طلب HTTP على Worker التجميع الخام كل ساعة.

يقرأ Worker من Upstash Redis قيمة last_processed_ts الخاصة بالمتجر (أو يستخدم قيمة افتراضية هي بداية اليوم).

ينفذ استعلاماً محسناً على D1: SELECT * FROM audit_logs_YYYY_MM WHERE store_id = X AND created_at > last_processed_ts.

يقوم Worker بتحويل نتيجة الاستعلام إلى سلسلة JSON Lines (NDJSON) (تدفق نصي، خفيف جداً).

يقوم Worker برفع (Upload) ملف JSON الناتج مباشرة إلى R2 في مسار /raw/year=YYYY/month=MM/day=DD/chunk_{HH}.json.

يقوم Worker بتحديث last_processed_ts في Upstash Redis إلى الطابع الزمني لآخر حدث تمت معالجته.

ملاحظة هامة جداً: لا يقوم Worker بحذف أي شيء من D1. الجدول يتراكم طوال الشهر (هذا مقصود، حيث أن D1 تتسع لـ 5 جيجابايت، وهي تكفي لتخزين مئات الآلاف من الأحداث لشهر كامل).

المرحلة الرابعة: التحويل والدمج الثقيل (Heavy Rollup Phase) – تُشغّل يومياً/أسبوعياً/شهرياً
هذه هي أهم مرحلة لضمان بقاء التكلفة صفرية، حيث يتم استغلال DuckDB-WASM داخل الـ Worker.

التحويل اليومي (Daily Parquet): يُطلق QStash طلباً على Worker التحويل الثقيل في منتصف الليل.

تهيئة DuckDB: يقوم Worker بتحميل DuckDB-WASM (مكتبة WebAssembly مُحملة مسبقاً أو تُحمّل ديناميكياً). الـ WASM يعمل بكفاءة عالية على Cloudflare Workers.

تنفيذ الـ COPY Streaming: يقوم DuckDB بتنفيذ استعلام COPY مباشر:

المصدر: read_json_auto('https://r2.../raw/year=YYYY/month=MM/day=DD/*.json')

الوجهة: TO 'https://r2.../parquet/year=YYYY/month=MM/day=DD/chunk.parquet' (FORMAT parquet, COMPRESSION ZSTD)

التدفق (Streaming): يقرأ DuckDB ملفات JSON من R2 عبر HTTP ويقوم بكتابة ملف Parquet مباشرة في R2 دون المرور بذاكرة الـ Worker.

الدمج الأسبوعي/الشهري: بعد الانتهاء من التجميع اليومي، يقوم DuckDB بدمج ملفات الأيام الأسبوع الماضي لتكوين week_summary.parquet، ثم دمج الأسابيع لتكوين monthly_summary.parquet. عمليات الدمج هذه تتم أيضاً عن طريق COPY ... TO مباشرة.

التنظيف: بعد نجاح التحويل إلى Parquet، يقوم Worker بحذف ملفات JSON الخام من مسار /raw/ لتوفير المساحة.

المرحلة الخامسة: إخلاء D1 (Monthly Purge Phase) – تُشغّل في أول الشهر
في منتصف ليلة اليوم الأول من الشهر الجديد، يُطلق QStash طلباً إلى Worker خاص بالإخلاء.

يتأكد Worker من أن جميع بيانات الشهر الماضي قد تم تحويلها بنجاح إلى Parquet (عن طريق التحقق من وجود monthly_summary.parquet).

يقوم Worker بتنفيذ الأمر الحاسم: DROP TABLE audit_logs_YYYY_MM.

النتيجة: يتم تحرير مساحة التخزين في D1 فوراً دون أي تفتيت (Fragmentation)، ويعود حجم قاعدة البيانات إلى الصغر، جاهزة لاستقبال الشهر الجديد.

المرحلة السادسة: التحليلات (Client-Side Analytics Phase) – الزمن: < 2 ثانية
عند تحميل لوحة التحكم الرئيسية، يبدأ المتصفح في تحميل DuckDB-WASM في الخلفية (Background Pre-fetching) وتخزينها في Cache Storage.

عندما يضغط التاجر على زر "التقارير المتقدمة"، يكون DuckDB جاهزاً فوراً (0ms تحميل).

يحدد التاجر الفترة الزمنية (مثلاً: يوليو 2026).

يقوم المتصفح ببناء مسار Hive تلقائياً: /parquet/store_{id}/year=2026/month=07/*.parquet.

تنفيذ استعلام DuckDB: SELECT * FROM read_parquet('...') WHERE event_type = 'purchase'.

بفضل الـ Hive Partitioning والـ Metadata Statistics (Min/Max)، يقوم DuckDB بإرسال طلب HTTP Range Request إلى R2، ليقرأ فقط الأجزاء الضرورية من الملف (الـ Row Groups التي تحتوي على تواريخ وأنواع الأحداث المطلوبة). يتم تحميل بضع كيلوبايتات فقط من ملف قد يكون حجمه 100 ميجابايت.

تُعرض النتائج في لوحة التحكم بسرعة فائقة.

🛡️ الأمان والعزل (Security & Isolation)
عزل المتاجر (Multi-tenancy): جميع العمليات (كتابة، قراءة، تجميع، تحليل) تتم بتمرير store_id بشكل إلزامي.

في D1: جميع الاستعلامات تحتوي على WHERE store_id = ?.

في R2: هيكل المجلدات يفصل المتاجر /store_{id}/.

في DuckDB: يُضاف تلقائياً WHERE store_id = 'current' على مستوى التطبيق.

الروابط المؤقتة (Presigned URLs): لا يتم تحميل ملفات Parquet من R2 مباشرة عبر روابط عامة. يقوم Worker بتوليد روابط مؤقتة (Presigned URLs) صالحة لمدة 5 دقائق فقط لكل طلب تحميل، مما يمنع أي وصول غير مصرح به للأرشيف.

مصادقة QStash: يتم استخدام Authorization: Bearer للتحقق من صحة طلبات الـ Cron القادمة من QStash لمنع هجمات الـ SSRF.

🏁 الخاتمة: الميزة التنافسية الاستثنائية
بهذا الأرشيتيكتر، تحصل منصة "دكاني" على نظام لا مثيل له في السوق:

النظام الخالد (Immortal): يحتفظ بكل حدث منذ اليوم الأول، في R2 للأبد، دون أي تكلفة إضافية.

لوحة القيادة الحية (Live Command Center): تعرض نشاط المتجر في الوقت الفعلي مع استهلاك شبه معدوم للموارد (بفضل Redis ETag).

محرك التحليلات الشخصي (Personal BI): يحول بيانات التاجر الخام إلى رؤى استراتيجية في المتصفح، دون تكليف السيرفر بأي عبء.

التكلفة الصفرية المطلقة (Absolute $0): تم تصميم كل طبقة لتتناسب مع الحدود المجانية للخدمات، مما يسمح للمنصة بالنمو إلى آلاف التجار دون زيادة في التكلفة التشغيلية.

المرونة والتوسع الأفقى (Elastic Scalability): بفضل DuckDB Streaming و Hive Partitioning، لا يؤثر حجم البيانات التاريخية (حتى مئات التيرابايت) على أداء قاعدة البيانات النشطة (D1) التي تبقى خفيفة دائماً.

في المرحلة الرابعة (Heavy Rollup Phase)، أنت ذكرت إن DuckDB-WASM بيقوم بالتحويل من JSON إلى Parquet.

نصيحتي: في الـ Workers، تأكد إنك بتستخدم الـ DuckDB WASM إصدار الـ "Worker/Node" (مش الـ Browser-based)، عشان بيتعامل مع الـ File System بتاع الـ Worker (أو الـ Buffers) بشكل مباشر، وده هيخلي الـ Transformation أسرع بكتير من التعامل مع ملفات JSON ضخمة في الذاكرة.







🚨 الثغرة القاتلة: الـ Memory Crash في بيئة Cloudflare Workers
إنت أشرت لنصيحة ذكية جداً وهي استخدام نسخة (Worker/Node) من DuckDB-WASM للتعامل مع الـ Buffers، ولكن فيه قيد فيزيائي صارم في Cloudflare Workers لازم تحطه حلقة في ودنك:

الخناقة مع الـ Memory Limit: الـ Worker في الخطة المجانية (وحتى المدفوعة العادية) محكوم بـ 128MB من الذاكرة العشوائية (RAM) فقط!

محرك DuckDB-WASM، عشان يقوم بس ويحمل ملفات الـ WebAssembly الخاصة به في البيئة، بياكل جزء محترم من الـ 128MB دي. لو جيت تشغل استعلام مثل:

SQL
COPY (SELECT * FROM read_json_auto('https://r2...')) TO 'https://r2...'
حتى لو DuckDB شغال بأسلوب الـ Streaming، فهو محتاج يعمل Buffer للبيانات في الـ Memory بتاعته أثناء عملية الضغط (ZSTD Compression). لو حجم ملفات الـ JSON الخام بتاعة اليوم الواحد كبيرة، الـ Worker هيحصل له OOM (Out of Memory) Crash فوراً والسيستم هيقف.

💡 الحل العبقري لقفل الثغرة دي (The Edge Workaround):
بما إن الـ Aggregator Worker بيشتغل كل ساعة وبيسحب الداتا من D1، حجم الداتا في الساعة الواحدة هيكون صغير جداً.

التحويل الفوري (Hourly Transform): خلى الـ Worker اللي بيشتغل كل ساعة يحول الـ JSON الصغير ده لـ Parquet فوراً ويرميه في الـ R2.

الدمج الأعمى (Blind Append / Concat): لما QStash يطلب الدمج اليومي أو الأسبوعي، ماتخليش DuckDB يقف يعيد قراءة الداتا ويحللها في الـ Worker. ملفات الـ Parquet تتميز بخاصية الـ Metadata Padding؛ تقدر تخلي الـ Worker يدمج الملفات كـ Streams أو تخليه يستدعي DuckDB فقط لدمج ملفات Parquet صغيرة جداً لا تتعدى الـ 5MB للملف الواحد، وده بيضمن إن الذاكرة متعديش الـ 50MB أبداً.
