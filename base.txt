🏛️ الأرشيتيكتر النهائي المتكامل لقاعدة البيانات الأساسية (Core DB) – الإصدار 2.2
📜 الديباجة: فلسفة "المحرك التشغيلي الخالد"
هذا الأرشيتيكتر يُمثل التصميم المعماري النهائي لقاعدة البيانات الأساسية (Core DB) لمنصة "دكاني". يُركز على إدارة البيانات النشطة (المنتجات، الطلبات، العملاء، المخزون) بمعزل تام عن نظام الأوديت والتحليلات. صُمم ليكون:

مقاوماً للانهيار (Fail-Proof): لا نقطة فشل واحدة، وآليات تعافي ذكية.

قابلاً للتوسع عضوياً (Organically Scalable): يبدأ بسيطاً وينمو مع المنصة.

صفر التكلفة التشغيلية (Zero Cost): ضمن حدود الخدمات المجانية لـ Cloudflare و Upstash.

قابلاً للمراقبة والصيانة (Observable): مع طبقات متكاملة للرصد والتنبيه.

المبادئ الحاكمة:

الفصل التام: لا علاقة لهذا النظام بنظام الأوديت.

العزل الفيزيائي: كل مجموعة من التجار في قاعدة D1 مستقلة.

التوزيع الذكي: توزيع متوازن مع إمكانية التوسع دون ترحيل بيانات حية.

المعالجة الهجينة: تفريق بين العمليات الحرجة (متزامنة) والثانوية (غير متزامنة).

Idempotency الشاملة: كل عملية يمكن تكرارها بأمان.

المراقبة الشاملة: رؤية كاملة لأداء النظام وصحته.

🧱 المكونات الأساسية
1. قاعدة البيانات النشطة (D1 – The Operational DB)
الخاصية	التفاصيل
النوع	SQLite مُدارة وموزعة عالمياً من Cloudflare (D1)
الهيكل	مرحلة MVP: قاعدة D1 واحدة. مرحلة النمو: 3-5 قواعد. مرحلة النضج: 10-50 قاعدة.
الحد الأقصى	50,000 قاعدة (في الخطة المدفوعة)
الدور	تخزين البيانات النشطة (منتجات، طلبات، عملاء، مخزون) مع دعم ACID Transactions.
سياسة الاحتفاظ	البيانات النشطة فقط (آخر 6-12 شهراً). البيانات القديمة تُؤرشف (اختياري).
2. طبقة الـ Idempotency المركزية
الخاصية	التفاصيل
الدور	منع تكرار أي عملية في النظام (API, Queue, Migrations, Payments, Inventory).
التقنية	جدول idempotency في D1 (أو قاعدة منفصلة للتوسع) مع TTL ديناميكي حسب نوع العملية.
سياسة TTL	العمليات الحرجة (دفع، مخزون) → 24 ساعة. العمليات العادية → 1 ساعة.
آلية Timeout	إذا بقيت العملية pending لأكثر من 30 ثانية، تُعتبر failed ويُسمح بإعادة المحاولة.
3. طبقة التوجيه (اللامركزية)
المكون	الدور	التقنية
التوجيه الموزع	كل Worker يوجه الطلبات بنفسه (بدون Router مركزي).	In-Memory Cache + Redis Cache + KV (مصدر دائم).
مصدر الهوية	JWT يحمل merchant_id فقط (لا يحمل database_id).	JWT (مشفر).
خريطة التوزيع	تخزين store_id → database_id في KV مع نسخة في Redis.	Cloudflare KV + Upstash Redis.
4. طبقة المعالجة (النموذج الهجين)
نوع العملية	الأمثلة	طريقة المعالجة	التقنية
حرجة (Critical)	التحقق من المخزون، الخصم، إنشاء الطلب، معالجة الدفع، تحديث حالة الطلب.	متزامن (Sync)	Worker يتصل بـ D1 مباشرة مع ACID Transaction.
ثانوية (Non-critical)	تسجيل Audit، إشعارات، تحديث إحصائيات، أرشفة، معالجة الصور.	غير متزامن (Async)	Worker يكتب في Queue.
5. طبقة إدارة الـ Schema (التطور التدريجي)
المكون	الدور	التقنية
Migration Orchestrator	تطبيق تغييرات Schema على جميع قواعد D1 تدريجياً.	Cloudflare Worker (Cron) + KV لتسجيل الحالة.
Schema Versioning	كل قاعدة تحتفظ بجدول schema_version لتتبع الإصدار الحالي.	D1 (جدول schema_version).
Progressive Rollout	تطبيق Canary (قاعدة واحدة أولاً) مع مراقبة لمدة ساعة، ثم التوسع.	KV لتسجيل نجاح/فشل كل قاعدة.
6. طبقة المراقبة والرصد (Monitoring & Observability)
الطبقة	الأداة	الغرض
المقاييس (Metrics)	Cloudflare Analytics + Workers Analytics Engine	تتبع زمن الاستجابة، معدل الأخطاء، عمق Queue، نسبة نجاح Cache.
السجلات (Logs)	سجلات منظمة (JSON) إلى R2 أو Workers Logs	تتبع كل طلب مع correlation_id.
التنبيه (Alerting)	Upstash Redis + Telegram Bot	تنبيهات فورية عند تجاوز العتبات (معدل أخطاء > 5%، عمق Queue > 80%).
7. خطة التعافي من الكوارث (Disaster Recovery)
المستوى	الإجراء	التكرار
1. Backup D1	نسخ احتياطي تلقائي لقاعدة D1 في R2.	يومياً.
2. Redundancy R2	نسخ ملفات Parquet (الأرشيف) إلى منطقة تخزين احتياطية.	فورياً.
3. Multi-Region	(للمستقبل) نشر Workers في مناطق متعددة.	—
8. استراتيجية الاختبار (Testing Strategy)
المستوى	الأداة	الغرض
Unit Tests	Vitest	اختبار دوال التوجيه، Idempotency، الـ Schema.
Integration Tests	Vitest + D1 Local	اختبار التفاعل بين Worker و D1 و Queue.
Load Tests	K6	محاكاة 1,000 طلب/ثانية، 10,000 تاجر.
Chaos Tests	استدعاء يدوي	قطع الاتصال بـ D1، إسقاط Worker، تأخير Queue.

🔄 تدفق البيانات التفصيلي
المرحلة 1: استقبال الطلب (Ingress)
العميل يرسل طلباً (مع X-Idempotency-Key في الـ Header إذا كان طلب كتابة).

Worker يستقبل الطلب (Sync أو Async حسب النوع).

التحقق من Idempotency: البحث في جدول idempotency.

إذا كان المفتاح موجوداً وحالته completed → إعادة النتيجة المخزنة (no-op).

إذا كان المفتاح موجوداً وحالته pending وأقدم من 30 ثانية → تحديث الحالة إلى failed والسماح بالمحاولة من جديد.

إذا كان المفتاح موجوداً وحالته pending ولم يمر 30 ثانية → رفض الطلب (لمنع التزامن).

إذا لم يكن المفتاح موجوداً → إنشاء سجل جديد بحالة pending.

Rate Limiting: التحقق من حدود التاجر و IP في Redis.

المعالجة:

حرجة (Sync): بدء ترانزاكشن في D1، تنفيذ العملية، إنهاء الترانزاكشن، تحديث Idempotency إلى completed، حذف Cache المرتبط، رد للعميل.

ثانوية (Async): كتابة الرسالة في Queue مع idempotency_key و sequence_number، رد للعميل بـ 202 Accepted.

المرحلة 2: المعالجة غير المتزامنة (Async)
Queue Consumer يقرأ دفعة من الرسائل (50 رسالة).

يفرز الرسائل حسب store_id، ثم حسب sequence_number (ضمن كل متجر).

لكل متجر، يعالج الرسائل بالتسلسل.

لكل رسالة:

التحقق من Idempotency مرة أخرى (في D1).

تنفيذ العملية (كتابة في D1، إرسال إشعار، إلخ).

تحديث Idempotency إلى completed.

Ack للرسالة (Message-by-Message Ack).

تسجيل Metrics (وقت المعالجة، النجاح/الفشل).

في حالة الفشل: إعادة المحاولة (حتى 3 مرات) مع تأخير تصاعدي، ثم نقل إلى DLQ (Dead Letter Queue).

المرحلة 3: القراءة (Read)
العميل يطلب بيانات (عرض المنتجات، حالة الطلب، إلخ).

Worker يستقبل الطلب (Sync).

استعلام Cache: البحث في Redis عن البيانات المطلوبة.

إذا وُجدت → إعادة البيانات (مع تحديث TTL إذا لزم الأمر).

إذا لم توجد → استخدام Mutex Lock (لمنع Cache Stampede):

الحصول على Lock (مع Retry Limited + Exponential Backoff).

جلب البيانات من D1.

تخزينها في Redis مع TTL مناسب.

تحرير Lock.

إعادة البيانات.

في حالة فشل Lock (بعد 3 محاولات)، جلب البيانات مباشرة من D1 (مع تحذير في السجلات).

المرحلة 4: إدارة الـ Schema (التطور التدريجي)
Migration Orchestrator يُشغّل أسبوعياً (أو عند الحاجة) بواسطة QStash.

يتحقق من schema_version في كل قاعدة D1.

يطبق التغييرات الجديدة على قاعدة واحدة أولاً (Canary).

يراقب الأداء والأخطاء لمدة ساعة.

إذا نجحت، يطبق على باقي القواعد تدريجياً (قاعدة تلو الأخرى).

إذا فشلت، يُجري Rollback فوراً ويُسجّل الفشل في KV لإعادة المحاولة لاحقاً.

يسجل الإصدار الجديد في جدول schema_version لكل قاعدة.
